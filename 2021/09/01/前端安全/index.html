XSS攻击

XSS攻击

XSS攻击，Cross-Site Scripting（跨站脚本攻击），为了和CSS区分，所以把第一个字母替换成了X。

XSS攻击是一种代码注入式攻击，攻击者通过对网页注入恶意代码，可以获取到用户的cookie、sessionID等敏感信息，进而危害数据安全。

XSS攻击的本质是，将恶意代码注入到网页中，从而和正常代码混淆在一起。浏览器无法区分哪些是正常代码，哪些是恶意代码而导致恶意代码被执行。

由于脚本在用户的终端被执行，这就导致了用户的隐私数据可以被脚本获取，正常来说，输入框的范围都是较短的，但是可以通过在代码中引用外链，将大量的外部Js代码引入到页面中并执行，这就会导致十分严重的后果。

XSS攻击的分类

存储型XSS

  1. 攻击者将恶意代码提交到数据库中；
  2. 服务端从数据库中将恶意代码取出，进行HTML拼接后返回给浏览器；
  3. 浏览器进行解析执行，恶意代码也被混入其中；
  4. 恶意代码窃取用户数据并发送到攻击者的网站，或冒充用户进行指定操作。

反射型XSS

  1. 攻击者构造出特殊的URL；
  2. 用户打开URL时，服务端将恶意代码取出，进行HTML拼接后返回给浏览器；
  3. 浏览器进行解析执行，恶意代码也被混入其中；
  4. 恶意代码窃取用户数据并发送到攻击者的网站，或冒充用户进行指定操作。

反射型XSS和存储型XSS的区别是：

存储型XSS将恶意代码存储在数据库中，而反射型XSS将恶意代码放在URL里，反射型XSS通常被用在通过URL传递参数，比如搜索、跳转等功能时，并且需要用户主动点击URL链接才会生效。

DOM型XSS

  1. 攻击者构造出特殊的URL；
  2. 用户打开带有恶意代码的URL；
  3. 用户浏览器接收到响应之后进行执行，前端JavaScript引擎取出URL中的恶意代码并执行；
  4. 恶意代码窃取用户数据并发送到攻击者的网站，或冒充用户进行指定操作。

DOM型XSS与前两种XSS的区别：

DOM型XSS是前端要注意的安全问题，而其他两种的恶意代码都是在服务端进行拼接后返回给浏览器。

XSS攻击的预防

XSS攻击主要有两个步骤：

  1. 攻击者提交恶意代码；
  2. 浏览器执行恶意代码。

当我们想拦截用户输入时，我们会遇到这样一个问题：不知道用户提交的内容要用在哪里。

对待不同情况的使用，是否进行转义，进行转义的规则要选择哪个，都是问题。

所以我们一般在第二个步骤进行防范XSS攻击：

防范存储型XSS和反射型XSS：

  1. 进行纯前端渲染；
  2. 转义HTML；

防范DOM型XSS：

  1. 不要把不可信的数据作为参数使用.innerHTML / .outerHTML / document.write()等；
  2. 避免将不可信的数据加入到onload() / onclick() / onerror()等API中；

其他XSS防范措施

  1. 进行Content Security Policy安全验证；
  2. 输入内容控制长度；
  3. HTTP-only Cookie；
  4. 验证码；
  5. …

CSRF攻击

CSRF攻击

CSRF（Cross-Site Request Forgery）Forgery: 伪造, 赝品跨站伪造请求：攻击者诱导受害者进入第三方网站后，在第三方网站中，利用用户已经登录的Cookie来冒充用户进行冒名操作来攻击。

链接类型的CSRF攻击

链接类型的CSRF攻击并不常见，要诱导用户主动点击链接之后才可以进行CSRF攻击。这类攻击通常在论坛中发布的图片中嵌入恶意链接，或者以广告的形式来诱导用户中招。

CSRF攻击的特点

  • 攻击一般发起在第三方网站，而不是被攻击的网站；
  • 攻击通过使用受害者的Cookie来冒充受害者进行操作，而不是直接窃取受害者的Cookie；

CSRF攻击的防护策略

  1. 设置cookie的SameSite属性

设置方法为：

Set-Cookie: sessionID = WangQing; SameSite=None; Secure

SameSite有三个value：

  • Strict：跨站点时，任何情况下都不会发送Cookie，只有当前网页的URL和请求目标一致，才会带上Cookie；
  • Lax：默认值，大多数情况也是不发送第三方Cookie，但是导航到目标网址的GET请求除外；
  • None：关闭SameSite属性。
  1. CSRF Token

服务器计算出Token后，前端将其存放在sessionStorage或localStorage中，每次请求都要带上。攻击者无法取得Token，这样可以校验出是否是第三方网站的请求；

  1. 服务端通过请求头的Referer和Origin字段，判断请求来源；

Referer：假如在www.baidu.com里有一个 <a src="www.google.com">，点击<a>后，在发送给www.google.com的请求Header中就会带有Referer: www.baidu.com；

Origin：指示了请求来源于哪个站点，比如Origin: https://developer.mozilla.org，除了不包含路径信息之外，其他的和Referer类似。

只有跨域请求，或者是同域发送POST请求时才会携带Origin请求头，而Referer除了

1. 通过本地文件打开网页；

2. 当前请求页面采用HTTP，来源网页采用HTTPS；

3. 直接输入网址或通过书签访问；

4. 使用Location.href / replace()；

5. 使用HTML5中的noreferrer

等情况下，只要浏览器能够获取到源地址，就会携带Referer请求头

当浏览器不能够获取源地址时，Referer请求头不会被携带，而Origin会在满足条件时就被携带，即使值为null。

  1. 手机、邮箱验证码等方式，对敏感操作进行二次验证；

在如今的许多主流网站中，跳转到外部链接的时候会提示“正在跳转到外部链接，安全性未知”，这也是从用户角度的一种CSRF防护。
