前端性能优化

发表于2021-08-20|更新于2021-08-20
  1. 减少 HTTP 请求

​

  1. 使用 HTTP/2

    解析速度快
    可多路复用
    首部压缩：首部表
    可设置高优先级
    流量控制
    服务器推送


  1. 使用服务端渲染
    • 客户端渲染先解析 HTML 文件，再向服务器请求资源，资源加载完毕后，开始渲染界面；
    • 服务端渲染在服务端就将 HTML 文件填充完毕，再返回给客户端。浏览器只需要解析 HTML 文件即可。

  1. 静态资源使用 CDN

    在多个位置部署服务器，用户离服务器越近，请求速度就越快。


  1. 将 CSS 文件放在头部，JavaScript 文件放在底部

    将 CSS 文件放在头部，防止用户看到无 CSS 样式的 HTML；
    将 JavaScript 文件放在底部，防止白屏现象，或者将script标签加上defer，异步下载，延迟执行。


  1. 使用iconfont代替图片图标

  1. 善用缓存，不重复加载相同的资源

  1. 压缩文件：使用webpack vite等

  1. 图片优化
    1. 图片 Lazy Load
    2. 响应式图片：根据屏幕大小自动加载合适的图片
    3. 调整图片大小：缩略图
    4. 降低图片质量
    5. 尽可能利用 CSS3 来代替图片：阴影、渐变等效果
    6. 使用 webp 格式的文件

  1. 通过webpack按需加载代码，提取第三方库代码，减少 ES6 转为 ES5 的冗余代码

  1. 减少重绘重排

    重排：当改变 Dom 元素位置或大小时，会导致浏览器重新生成渲染树；
    重绘：当重新生成渲染树后，就要将渲染树的每个节点重新绘制到屏幕上。但是并不是所有的操作都会导致重排，比如更改字体颜色，只会导致重绘。
    重排会导致重绘，但是重绘不一定会导致重排。


    **重绘和重排是非常宝贵的，因为 JavaScript 引擎线程和 GUI 渲染线程不能同时工作。**
    **会导致重排的操作：** 1. **添加或删除可见的 Dom 元素** 1. **元素位置或尺寸改变** 1. **内容改变** 1. **浏览器窗口尺寸改变**

    > **如何减少重排重绘？** > 1. **用 JavaScript 修改样式时，尽量不要直接写样式，而是用替换 class；** > 1. **如果要对 Dom 元素执行一系列操作，可以将 Dom 元素脱离文档流，修改完成后再将其带回文档。推荐使用隐藏元素**`display: none`**或文档碎片**`DocumentFragement`**。**
  1. 使用事件委托

    利用事件冒泡，只需指定一个事件处理函数，就可以管理某一类型的所有事件。

  1. 注意函数的局部性

像以上这种情况，如果使用if else，最多要判断 7 种情况，而switch只需要进行一次判断。

    当条件语句特别多时，switch和if else不是最佳的选择，这时可以使用数组、对象、Set或Map来构建。

  1. 避免页面卡顿
    • 60 FPS 和设备刷新率

实际上，因为浏览器还有一些整理工作要做，所以每一帧所需的工作需要在 10ms 内完成，如果无法符合该预算，帧率则会下降，内容会在屏幕上抖动，此现象通常被称之为卡顿。


**对于一些长时间运行的 JavaScript，我们可以使用定时器进行切分，延迟执行。**
  1. 使用requestAnimationFrame来实现视觉变化

  1. 使用 Web Workers

  1. 使用位操作

  1. 不要覆盖原生方法

    无论如何优化，都比不上原生方法。原生方法是用低级语言写的，可以被编译成机械码，成为浏览器的一部分。


  1. 减低 CSS 选择器的复杂性（可选）
    • 浏览器读取选择器，是按从右到左的顺序进行读取
  • CSS 选择器优先级

内联样式 > ID 选择器 > 类选择器 > 标签选择器

可以得出结论：

  • 选择器越短越好
  • 尽量使用高优先级的选择器，例如 ID 和类选择器
  • **避免使用通配符 ***

因为 CSS 最慢和最快的选择器性能差别非常小，因此没有优化的必要。

  1. 使用flexbox而不是较早的布局模型

  1. 使用transform和opacity属性更改来实现动画

    在 CSS 中，transform和opacity两个属性更改不会导致重排和重绘，它们是可以由合成器composite单独处理的属性
