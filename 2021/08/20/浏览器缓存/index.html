浏览器缓存

发表于2021-08-20|更新于2021-08-20

浏览器缓存就是指在本地使用的计算机中开辟一个内存区，同时也开辟一个硬盘区作为数据传输的缓冲区，然后用这个缓冲区来暂时保存用户以前访问过的信息。

浏览器缓存一般分为：强缓存、协商缓存。
浏览器缓存位置一般分为四类：Service Worker => Memory Cache => Disk Cache => Push Cache

强缓存

强缓存就是当我们在访问 URL 时，不会向服务器发送校验请求，而是直接从缓存中读取数据，但是还是会返回 200 的状态码。

如何设置强缓存？

我们第一次进页面，进行请求的时候，浏览器会根据response Header来判断是否对资源进行缓存，如果响应头中有 expires pragma cache-control 字段，代表这是强缓存，浏览器就会把资源存在 Memory Cache 或者 Disk Cache 中。


第二次请求中，浏览器判断请求参数，如果符合强缓存条件就直接返回 200 状态码，从本地缓存中请求数据。否则把相应参数存在 request header 中，看是否符合协商缓存，符合则返回状态码 304，不符合则服务器返回全新资源。

Status 304：Not Modified 未修改
指的是浏览器所请求的资源未修改。

expires

是指 HTTP1.0 控制网页缓存的字段，值为一个时间戳，准确来讲是格林尼治时间，服务器返回该请求结果缓存的到期时间，意思就是，再次发送请求时，如果未超过过期时间，直接使用该缓存，如果过期了则重新请求。


但是expires有一个缺点，就是它判断是否过期是根据计算机的本地时间来判断的，本地时间是可以被修改的。
Cache-Control
是指 HTTP 1.1 控制网页缓存的字段，当 Cache-Control 和 expires 同时存在时，Cache-Control优先级更高，
Cache-Control的value有：

  • public：浏览器和服务器都可以缓存资源；
  • privite：只有浏览器可以缓存资源；
  • no-cache：浏览器缓存资源，但是是否缓存需要经过协商缓存来验证；
  • no-store：不使用缓存；
  • max-age：缓存保质期。


Cache-Control使用了max-age相对时间，解决了expires时间可能不同步的问题。

pragma

这是 HTTP1.0 中禁用网页缓存的字段，其取值为no-cache，和Cache-Control中的no-cache效果一样。

缓存位置

如上所说，强缓存我们会把资源放到 Memory Cache 或 Disk Cache 中，那么，什么资源放在 Memory Cache 中，什么资源放在 Disk Cache 中？

存储图像和网页等资源主要缓存在 Disk Cache，操作系统缓存文件等资源大部分都会缓存在 Memory Cache 中。
具体由浏览器分配，看谁的资源利用率不高就分给谁。
​

我们可以看到，Memory Cache 请求时间都是 0 ms，我们来梳理一下。
​

查找浏览器缓存时会按照顺序查找：Service Worker => Memory Cache => Disk Cache => Push Cache。

1. Service Worker

Service Worker 是运行在浏览器背后的独立线程，一般可以用来实现缓存功能。使用 Service Worker 的话，传输协议必须是 HTTPS。因为 Service Worker 中涉及到请求拦截，必须使用 HTTPS 来保证安全。Service Worker 的缓存与浏览器其他内建的缓存机制不同，它可以让我们自由控制缓存哪些文件、如何匹配缓存、如何读取缓存，并且缓存是持续性的。

2. Memory Cache

内存中的缓存，主要包含的是当前页面中已经抓取到的资源，例如页面上已经下载的样式、脚本、图片等。读取内存中数据的速度肯定比硬盘快，内存缓存虽然读取高效，但是持续时间会很短，会随着进程的释放而释放。一旦我们关闭 Tab 页面，内存中的缓存也就被释放了。

3. Disk Cache

硬盘中的缓存，读取速度慢点，但是什么都能存储到硬盘之中，和 Memory Cache 相比在储存容量和时效性上有优势。
在所有浏览器缓存中，Disk Cache 的覆盖面基本是最大的。它会根据 HTTP Header 中的字段判断哪些资源需要缓存，哪些资源可以不请求直接使用，哪些资源已经过期需要重新请求。并且即使在跨站点的情况下，相同地址的资源一旦被 Disk Cache 缓存下来，就不会再次去请求数据。绝大部分的缓存都来自于 Disk Cache。
​

而 Memory Cache 相比 Disk Cache 要快得多。
举个栗子：
从远程服务器直接提取访问文件可能需要 500 ms，那么从 Disk Cache 访问可能只需要 10 - 20 ms，而 Memory Cache 只需要 100 ns，更高级的还有 L1 缓存访问（最快的和最小的 CPU 缓存），只需要 0.5 ns。

很神奇的，我们又看到了一个 prefetch cache，这个又是什么呢？

Prefetch Cache（预取缓存）

fetch：取、拿

在 link 标签上带上了 prefetch，再次加载时候就会出现 Prefetch Cache。
​

Prefetch Cache 是预加载的一种方式，被标记为 Prefetch 的资源，将会被浏览器在空闲时间加载。

4. Push Cache（推送缓存）

Push Cache 是 HTTP/2 中的内容，当以上三种缓存都未命中时，它才会被使用。它只在会话（Session）中存在，一旦会话结束就会被释放，并且缓存时间也很短暂，在 Chrome 浏览器中只会存在 5 min 左右，同时它也并非严格执行 HTTP 请求头中的缓存指令。

协商缓存

协商缓存就是强缓存失效之后，浏览器携带缓存标识向服务器发送请求，由服务器根据缓存标识来决定是否使用缓存的过程。
​

主要有以下两种情况：

  • 协商缓存生效：Status 304

  • 协商缓存失效：Status 200 和请求结果

如何设置协商缓存？

Last-Modified

modified：修改的

Last-Modified 是服务器响应请求时，返回该资源文件在服务器最后被修改的时间。

If-Modified-Since 是浏览器再次发起请求时，携带上次请求返回的 Last-Modified，通过此字段告诉服务器上次请求返回的最后被修改时间。服务器收到请求后，如果检测到 If-Modified-Since 字段，则会对 value 与服务器上资源最后被修改的时间进行比较，如果服务器上的资源被修改的时间晚于 value，则返回新的资源，并带上 Status 200；否则返回 Status 304，表示资源未过期，可以继续使用。

Etag

Etag 是服务器响应请求时，返回当前资源文件的一个唯一标识（由服务器生成）。

If-None-Match 是浏览器再次向服务器发送请求时，带上上次请求得到的 Etag，通过此字段告诉服务器该资源上次请求返回的唯一标识值。服务器检测到 If-None-Match 字段后，将 value 与服务器上该资源的 Etag 值做对比，一致则返回 Status 304，否则返回新的资源，并带上 Status 200。

Etag / If-None-Match 优先级高于 Last-Modified / If-Modified-Since。

缓存方案

目前的项目大多数都使用这种缓存方案：

  • HTML：协商缓存；
  • css、js、图片：强缓存，文件名带上 Hash 值。

    为什么要带上 Hash 值：如果服务器上的资源发生改变，浏览器请求该资源时，则会发现路径错误，从而得到新的资源。

强缓存与协商缓存的区别

  1. 强缓存不会发送请求到服务器，所以有的时候资源已经更新了但是浏览器还不知道；而协商缓存每次都会发送请求到服务器，所以资源是否更新，浏览器是知道的；
  2. 大部分 web 服务器都默认开启协商缓存。

刷新对于强缓存和协商缓存的影响

  1. 当使用 ctrl + F5 强制刷新网页时，直接从服务器进行加载，跳过强缓存和协商缓存；
  2. 当使用 F5 刷新网页时，跳过强缓存，但是会检查协商缓存；
  3. 当浏览器地址栏中直接写入 URL 时，浏览器中会发现缓存中已经有这个文件了，不用继续进行请求，而是直接从缓存中取到。
